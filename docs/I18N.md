# 다국어(i18n) 확장 설계

> 상태: 🔨 **기반 구현됨** (2026-07-21). 1단계 인프라 완료 — `src/i18n/`
> (`config.ts` 로케일 목록·기본값, `messages/ko.ts` UI 사전, `getDictionary()`),
> 공유 클라이언트 크롬(모바일 내비·검색)이 사전을 사용합니다. **남은 것**은
> `[lang]` 라우팅 도입과 실제 대상 로케일의 **콘텐츠 번역**이며, 후자는 커피
> 도메인 정확성이 필요해 사용자가 번역을 제공·승인해야 진행합니다(임의 기계번역
> 지양). 라우팅만 먼저 넣으면 번역 없는 `/en`이 반쪽이 되므로 대상 로케일 확정
> 시 함께 진행합니다. 아래는 전체 설계입니다.

## 목표

- 기본 언어(한국어)는 지금처럼 `/`에서 서비스.
- 추가 언어는 하위 경로(`/en/...`)로 확장. 페이지 로직 대부분은 재사용.
- SEO를 위해 언어별 대체 링크(hreflang)와 사이트맵 제공.

## 라우팅 방식

Next 16 App Router에서 선택지:

| 방식 | 예 | 판단 |
| --- | --- | --- |
| 하위 경로 세그먼트 `/[lang]/...` | `/en/wiki/...` | **채택** — 정적 생성과 잘 맞고 단순 |
| 도메인 분리 | `en.bean-wiki…` | 운영 복잡, 현시점 과함 |

권장: `src/app/[lang]/...`로 라우트를 감싸고 `generateStaticParams`로
`["ko","en"]`를 생성. 기본 언어 `ko`는 `/` 리라이트(또는 `/ko` 정규화 후 리다이렉트)로
기존 URL을 유지.

## 콘텐츠 구조

현재: `src/content/articles/*` 문서당 파일 하나(한국어).

제안: 언어별 콘텐츠를 분리하고 파사드가 로케일을 받도록 확장.

```
src/content/
  ko/articles/*, ko/glossary.ts, ko/categories.ts
  en/articles/*, en/glossary.ts, ...
```

파사드 API를 로케일 인자로 확장(기존 시그니처는 `ko` 기본으로 유지):

```ts
getArticle(slug, locale = "ko")
articlesByCategory(name, locale = "ko")
// ...
```

`slug`와 `category.slug`는 언어 공통(영문 slug)으로 두고, 표시 문자열만
언어별로 다르게 하면 URL이 안정적입니다.

## UI 문자열

현재 컴포넌트에 한국어 문자열이 하드코딩되어 있습니다(예: "모든 문서",
"최근 검색", "이 문서 공유"). 이를 언어별 사전으로 추출:

```
src/i18n/messages/ko.ts, en.ts   // { "nav.docs": "문서" / "Docs", ... }
```

경량 사전 + 헬퍼(`t(key, locale)`)로 충분하며, 규모가 커지면 `next-intl` 도입 검토.

## 메타데이터·SEO

- `layout`의 `<html lang>`을 현재 로케일로.
- 각 페이지 `alternates.languages`(hreflang)로 언어별 대체 URL 제공.
- `sitemap.ts`를 로케일 순회로 확장, 각 URL에 `alternates` 추가.
- `SITE_URL` 상수는 그대로 재사용.

## 단계별 이행

1. **UI 문자열 추출** — 하드코딩 문자열을 `ko` 사전으로 옮기고 `t()`로 대체
   (동작 변화 없음, 리팩터링만). 이 단계가 가장 큰 사전 작업.
2. **파사드 로케일화** — 콘텐츠를 `ko/`로 이동, 헬퍼에 `locale` 인자 추가(기본 ko).
3. **라우트 래핑** — `app/[lang]` 도입, `ko` 기본 경로 유지, `generateStaticParams`.
4. **영어 콘텐츠 추가** — 문서 단위로 점진 번역, 없는 문서는 언어 폴백 처리.
5. **hreflang + 사이트맵** 언어별 확장.

## 리스크·판단

- 큰 작업이므로 1단계(문자열 추출)만 먼저 해도 이후가 수월해짐 — 우선 착수 후보.
- 콘텐츠 번역은 분량이 커서 자동화보다 문서 단위 점진 확장이 현실적.
- slug를 언어 공통(영문)으로 유지해야 URL·연관 링크·태그 구조가 언어에 걸쳐 일관됨.
