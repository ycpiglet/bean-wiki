# Bean Wiki 편집 기능 설계 — "진짜 위키" 만들기

> 상태: 설계(제안). 2026-07-22 딥리서치(MediaWiki·나무위키·현대 위키 앱·Git-CMS·tag_manual 에디터·현 코드베이스 제약) 기반.
> 목표: 브라우저에서 직접 문서를 쓰고 고치고, 사진을 찾아 붙이고, 문서끼리 `[[링크]]`로 엮이는 — 나무위키/위키피디아처럼 **살아있는 위키**.

## 1. 사용자 시나리오 (요구사항)

1. 모든 문서 페이지에 **[편집]** 버튼 → WYSIWYG 에디터로 그 자리에서 수정 → 저장.
2. 새 문서 작성: 붉은 링크(없는 문서로의 링크) 클릭 or "새 문서" 버튼.
3. **이미지**: 검색(Unsplash/Wikimedia)해서 바로 삽입하거나 파일 업로드/붙여넣기.
4. **검색창이 모든 페이지에** 상주 (현재는 홈에만 있음).
5. **하이퍼링크 상호 지원**: 본문 안 `[[문서명]]` 링크, 자동완성, 역링크(이 문서를 가리키는 문서들).
6. 초기 편집자는 소유자 1인. 나중에 기여자 확장 가능해야 함.

## 2. 리서치에서 채택하는 것 (근거)

| 출처 | 채택 |
|---|---|
| **tag_manual** (사용자의 기존 에디터) | 검증된 UX 패턴 전부: 슬래시 명령(+메뉴), 4색 콜아웃/노트/토글 블록, 이미지 드래그·붙여넣기 즉시 업로드+크기 프리셋(50/기본/전체), img-row 다중 배치, **localStorage 초안 자동저장(900ms 디바운스) + 명시적 저장/스냅샷 구분**, 소스 보기 패널, 버전 패널. 기술적으론 TinyMCE였으나 한계(드래그 재배치 불가, iframe 미리보기)도 확인됨 |
| **MediaWiki/위키피디아** | 붉은 링크(없는 문서 → 생성 유도), basetimestamp 기반 **편집 충돌 감지**(락 없이), 파일 메타데이터(라이선스·출처·업로더) 문서화, 편집 요약(edit summary), 카테고리=본문 속 선언 |
| **나무위키/나무마크** | `[[문서#앵커]]` 섹션 링크, `#redirect` 리다이렉트, 상단 목차·하단 분류 배치, 각주 문화, 문서별 토론 분리, **툴바로 문법 진입장벽 낮추기**, 모든 페이지 검색 상주 |
| **현대 위키 앱** (Notion/Outline/Obsidian) | **TipTap**(ProseMirror·React) 또는 BlockNote가 현 세대 표준, `/`슬래시 명령이 최고 효과 UX, `[[` 입력 시 퍼지 자동완성(Obsidian), **빌드타임 backlinks.json**(런타임 비용 0), 붙여넣기·드래그만으로 이미지 |
| **Git-CMS 리서치** | 기성 CMS(Sveltia/Keystatic)도 가능하나 우리 스키마(`## 제목 {#id}` 섹션·points 규약)와 "문서 페이지에서 바로 편집" UX엔 **커스텀 에디터 + GitHub API 커밋**이 적합. 이미지는 저장소가 아닌 **Vercel Blob**(저장소 비대화 방지). 라이브 반영 지연(1~2분)은 위키 편집 주기에 허용 가능 — DB 하이브리드는 필요해질 때만 |

## 3. 핵심 아키텍처 결정

### 결정: **인앱 TipTap 에디터 + Git-as-Database(GitHub API 커밋) + Vercel 자동 재배포**

```
[문서 페이지] --편집--> /edit/[slug] (TipTap WYSIWYG)
     ↓ 저장(편집 요약 포함)
[/api/edit] 검증(check-content 로직 공유) → 마크다운 직렬화(content-md 공유)
     ↓ Octokit(서버 PAT)으로 ko/en .md + order.json 커밋
[GitHub main] → Vercel 자동 빌드(~1-2분) → 라이브 반영
     ↑ 문서 역사 = git 히스토리 (공짜 버전관리·diff·복원)
```

**왜 이 조합인가**
- **현 자산 보존**: 마크다운 소스, `check-content` 무결성 게이트, ko/en 구조, 74페이지 정적 성능·SEO가 전부 그대로. 편집은 "파일을 고치는 다른 손"일 뿐.
- **버전 관리 공짜**: 모든 편집이 git 커밋(편집 요약 = 커밋 메시지). 역사/diff/복원 UI는 GitHub API로 읽기만 하면 됨.
- **tag_manual 대비**: Supabase 스택 대신 이미 있는 GitHub+Vercel만 사용. 에디터 UX 패턴은 그대로 이식하되, 라이브러리는 TinyMCE → **TipTap** 권장(React 네이티브, `[[위키링크]]` 커스텀 노드·자동완성 확장이 자연스러움, 드래그 재배치 가능 — TinyMCE에서 막혔던 부분).
- **대안 비교**:

| 방식 | 장점 | 단점 | 판단 |
|---|---|---|---|
| 기성 Git-CMS (Sveltia/Keystatic) | 빨리 붙임 | 별도 /admin UX(위키답지 않음), 섹션 `{#id}` 스키마 맞춤 한계 | 보류 |
| 런타임 DB (Supabase/Neon)+ISR | 초 단위 반영 | 정적 파이프라인 재설계, 정합성 게이트 상실, git 역사 분리 | 지연이 실제로 아플 때 하이브리드로 |
| **커스텀 에디터+GitHub 커밋** | 위키 UX·자산 보존·버전 공짜 | 반영 1~2분, 직렬화 왕복 정밀도 필요 | **채택** |

- **반영 지연 완화**: 저장 직후 에디터가 "게시 중" 배너 + 낙관적 로컬 미리보기 제공. Deploy Hook으로 빌드 즉시 트리거. (그래도 아프면 Phase 4에서 DB 하이브리드.)

### 에디터 ↔ 콘텐츠 스키마 매핑

현 Article 구조가 곧 에디터의 블록 제약이 된다 (스키마가 강제되는 WYSIWYG):

| 에디터 블록 | 마크다운 |
|---|---|
| H2 헤딩(앵커 자동 부여) | `## 제목 {#id}` |
| 문단 | 빈 줄 구분 문단 |
| 불릿 목록(섹션당 1개) | `- ` points |
| `[[문서명]]` 인라인 노드 | `[[slug]]` (본문 내 신규 문법) |
| 이미지 figure(출처·라이선스 포함) | `![캡션](url "attribution")` (신규) |
| 프론트매터 폼(제목·요약·분야·난이도·태그·fact) | frontmatter |

직렬화는 기존 `scripts/content-md.mjs`를 **공유 라이브러리로 승격**(`src/lib/content-md.ts`)해 에디터·API·빌드가 같은 코드로 왕복 — 마이그레이션 때 검증한 왕복 동등성 게이트를 저장 시마다 실행.

## 4. 단계별 기능 설계

### Phase 1 — 편집의 뼈대 (MVP)
- **전역 헤더 + 상주 검색**: 공용 `SiteHeader` 컴포넌트로 6개 페이지 헤더 통합, 컴팩트 검색창 상주(⌘K 유지). 검색 인덱스는 빌드타임 `search-index.json`(ko/en)으로 추출해 어느 페이지든 동일 사용.
- **인증**: `/edit/*`·`/api/edit/*`를 세션 쿠키로 보호. 로그인 = `EDIT_PASSWORD` env 검증(1인 운영 최소). 확장 시 GitHub OAuth 허용목록으로 교체 가능한 구조.
- **편집 페이지** `/edit/[slug]`: TipTap 에디터, 프론트매터 폼, tag_manual식 **localStorage 초안 자동저장**, 소스(마크다운) 보기 패널, 편집 요약 입력.
- **저장 API**: 검증(공유 check-content) → 직렬화 → Octokit 커밋(`content: edit(slug): <요약>`) → "게시 중" 상태 반환. **basetimestamp 충돌 감지**: 편집 시작 시점의 파일 SHA를 함께 보내 다르면 경고+diff.
- **새 문서** `/edit/new`: 스캐폴드 폼(slug·분야·난이도) → 본문 작성 → 커밋이 `order.json`도 갱신.

### Phase 2 — 위키다움 (상호 링크·이미지)
- **`[[위키링크]]`**: 본문 문법 추가. `[[` 입력 시 문서 퍼지 자동완성(제목/슬러그). 렌더러가 문단 문자열의 `[[slug]]`/`[[slug#앵커]]`를 Link로 변환. **붉은 링크**: 대상 부재 시 붉게 표시 → 클릭하면 그 slug로 새 문서 작성 진입.
- **역링크(Backlinks)**: 빌드타임에 전 문서의 위키링크를 파싱해 `backlinks.json` 생성 → 문서 하단 "이 문서를 참조하는 문서" 섹션. `check-content`는 깨진 위키링크를 **경고**(실패 아님 — 붉은 링크는 기능).
- **이미지**:
  - 업로드/붙여넣기/드래그 → `/api/upload` → **Vercel Blob** → 공개 URL 삽입 (tag_manual의 크기 프리셋·img-row 이식).
  - **검색 삽입**: `/image` 슬래시 명령 → Unsplash + Wikimedia Commons 검색 패널. **출처·작가·라이선스를 figure에 필수 저장**하고 캡션에 자동 표기 (라이선스 준수).
- **리다이렉트**: 문서 이름 변경 시 `redirects.json`에 구슬러그→신슬러그 기록, `next.config` redirects()로 301. 링크 파괴 방지.

### Phase 3 — 신뢰와 협업 기반
- **문서 역사 UI**: `/wiki/[slug]/history` — GitHub API로 해당 파일 커밋 목록(요약·시각·작성자), diff 뷰(diff-match-patch), "이 버전으로 복원"(역커밋).
- **ko/en 동기화 보조**: ko 저장 시 en 대응 문서의 섹션 구조가 어긋나면 에디터가 알림 + en 편집 바로가기 (지금은 check-content가 빌드에서만 잡음).
- **초안 상태**: frontmatter `draft: true` → 빌드는 하되 목록·사이트맵·RSS 제외. 게시 전 검토 흐름.
- **토론(선택)**: 문서별 GitHub Discussions/Issues 연결(나무위키식 분리 원칙, 자체 구현 없이).

### Phase 4 — 필요해지면 (미리 안 함)
- 편집 반영 초 단위가 필요해지면: Neon/Supabase 하이브리드(라이브 레이어)+ISR, git은 역사 원장 유지.
- 기여자 다수: GitHub OAuth 역할, 문서별 권한(나무위키 ACL 축소판), 편집 대기열(PR 기반 검토).
- 실시간 협업(CRDT)은 다중 동시 편집이 실제로 생기기 전까지 배제.

## 5. 크로스커팅 사항 (완결성 검토에서)
- **이미지 라이선스**: 검색 삽입 시 `{src, author, license, sourceUrl}`을 반드시 저장·표기. Unsplash는 API 규약상 다운로드 트리거+크레딧, Commons는 라이선스별 표기 준수.
- **오프라인**: 서비스 워커는 읽기 전용 유지. 편집 중 오프라인은 localStorage 초안이 방어(오프라인 커밋 큐는 만들지 않음 — 충돌 복잡도 대비 가치 낮음).
- **RSS/사이트맵**: 커밋→재빌드 구조라 자동 갱신(현 파이프라인 그대로). RSS에 이미지 enclosure는 이미지 도입 시 추가.
- **모바일 편집**: TipTap은 모바일 대응. Phase 1에선 데스크톱 우선, 모바일은 읽기+간단 수정 수준 목표.

## 6. 구현 순서·규모 (예상)

| 단계 | 내용 | 규모 |
|---|---|---|
| 1a | SiteHeader 통합 + 전역 검색(인덱스 추출) | M |
| 1b | 인증(세션) + /edit/[slug] TipTap + 직렬화 공유화 | L |
| 1c | 저장 API(검증→커밋) + 충돌 감지 + 새 문서 | M |
| 2a | [[위키링크]] 파서·렌더러·자동완성 + 붉은 링크 + backlinks | M |
| 2b | 이미지 업로드(Blob)+검색(Unsplash/Commons)+라이선스 | M |
| 2c | 리다이렉트 | S |
| 3 | 역사/diff/복원 UI, ko/en 동기화 보조, 초안 | M~L |

의존성 추가(모두 무료 티어 내): `@tiptap/react`+확장, `octokit`, `@vercel/blob`, `diff` 계열. 신규 env: `EDIT_PASSWORD`(또는 OAuth), `GITHUB_CONTENT_TOKEN`(fine-grained PAT, contents:write), `BLOB_READ_WRITE_TOKEN`, `UNSPLASH_ACCESS_KEY`(선택).

## 7. 결정이 필요한 것 (착수 전 확인)

1. **에디터 라이브러리**: 권장 = TipTap (React 네이티브·위키링크 확장·드래그 재배치). tag_manual과 동일한 TinyMCE로 가면 익숙함은 얻지만 위키링크 자동완성·블록 재배치에서 같은 한계 재현.
2. **반영 지연 1~2분 허용 여부**: 허용 → 위 설계 그대로. 초 단위 필수 → Phase 4 하이브리드를 앞당김(규모 +L).
3. **이미지 검색 소스**: Unsplash(고품질·상업 사용 용이) / Wikimedia Commons(커피 도판·라이선스 다양) / 업로드만. 복수 선택 가능.
